# LLM Chat & Context Management

**Depends on:** [LLM Integration](2026-03-18-llm-integration.md) (Milestones 1–4 must be complete)

## Overview

Add a chat sidebar to tiddly.me where users can converse with an LLM about their content, and integrate selection-based actions (improve, summarize, explain) that route through chat rather than standalone UI. This plan covers the context management strategy that underpins both features — how much content to inject, how to manage conversation length, and how to control token spend.

**Prerequisites from LLM Integration plan:**
- `LLMService` with streaming support (`stream()` method)
- `/ai/` router with auth, rate limiting, BYOK header extraction
- AI rate limits enforced per tier
- Cost tracking via Redis + daily cron flush

**Key decisions:**
- **Chat is per-item** — the current bookmark/note/prompt is the context. Global/cross-item chat (RAG) is a future feature.
- **Conversations are ephemeral** — not persisted to DB, lost on navigation/refresh. This keeps the scope manageable and avoids a new data model.
- **Selection actions route through chat** — the editor command menu (Cmd+/) triggers a chat message rather than a standalone popover, giving users the ability to iterate.
- **Token-budget-based context management** — rather than arbitrary character limits, we allocate a token budget and fill it intelligently.

- LLM Integration plan: `docs/implementation_plans/2026-03-18-llm-integration.md`

## Milestone 1: Context Management Strategy

### Goal & Outcome

Define and implement the backend logic for managing what context gets sent to the LLM and how conversation history is bounded. This is foundational — both chat and selection actions depend on it.

### The Problem

Every LLM call has a finite context window and a cost proportional to tokens sent. We need to decide:
1. How much of the current item's content to include as context
2. How to manage conversation history as it grows
3. How to prevent a single long note from blowing the token budget

### Design

#### Token budget allocation

Each chat request has a total token budget determined by the model's context window. We divide it into three fixed-priority tiers:

```
Total context window (e.g. 1M tokens for Gemini 2.5 Flash)
├── System prompt         — fixed, small (~500 tokens)
├── Item context          — up to a configurable cap (default: 8,000 tokens)
├── Conversation history  — fills remaining budget, oldest messages dropped first
└── Reserved for response — configurable (default: 4,000 tokens)
```

The key insight: **item context has a hard cap, conversation history is elastic.** A 50,000-token note doesn't get 50,000 tokens of context — it gets truncated to the cap. This keeps costs predictable and leaves room for conversation.

#### Item context injection

The current item's content is injected as a system message at the start of each request. Truncation strategy:

- If content fits within the item context cap: include all of it
- If content exceeds the cap: include the first N tokens up to the cap, with a note appended: `[Content truncated — showing first ~{n} characters of {total}]`

This is intentionally simple. Smarter strategies (semantic chunking, embedding-based retrieval of relevant sections) are future optimizations that belong in a RAG-focused plan.

#### Conversation history management

The backend receives the full conversation history from the frontend (since conversations are ephemeral and client-side). Before sending to the LLM:

1. Calculate tokens used by system prompt + item context + response reservation
2. Remaining budget = total budget - above
3. Walk conversation history from newest to oldest, accumulating token counts
4. Drop the oldest messages that don't fit
5. Always keep the most recent user message (the one being responded to)

If even the most recent user message + item context exceeds the budget, truncate the user message with a note.

#### Token counting

Use LiteLLM's `token_counter(model, text)` for accurate per-model token counts. This is a local operation (no API call) — it uses the model's tokenizer. Call it when building the request, not on every keystroke.

#### Configuration

```python
# New settings in config.py
llm_context_item_max_tokens: int = 8000       # Max tokens for item content
llm_context_response_reserved_tokens: int = 4000  # Reserved for LLM response
```

These are conservative defaults. With Gemini 2.5 Flash's 1M context window we have massive headroom, but keeping item context bounded means costs stay predictable even if someone has a 200-page note.

### Implementation Outline

#### 1. Context builder (`backend/src/services/llm_context.py`)

```python
class ContextBuilder:
    """Builds the messages array for an LLM chat request, managing token budgets."""

    def __init__(self, model: str, settings: Settings):
        self.model = model
        self.item_max_tokens = settings.llm_context_item_max_tokens
        self.response_reserved = settings.llm_context_response_reserved_tokens

    def build_messages(
        self,
        system_prompt: str,
        item_content: str | None,
        conversation: list[ChatMessage],
    ) -> list[dict]:
        """Returns a messages array that fits within the model's context window."""
        messages = []

        # 1. System prompt (always included)
        system_text = system_prompt
        if item_content:
            truncated_content = self._truncate_to_tokens(item_content, self.item_max_tokens)
            system_text += f"\n\n## Current Item Content\n\n{truncated_content}"
        messages.append({"role": "system", "content": system_text})

        # 2. Calculate remaining budget
        system_tokens = self._count_tokens(system_text)
        remaining = self._model_context_limit() - system_tokens - self.response_reserved

        # 3. Fill conversation history newest-first
        included = []
        for msg in reversed(conversation):
            msg_tokens = self._count_tokens(msg.content)
            if msg_tokens <= remaining:
                included.append(msg)
                remaining -= msg_tokens
            else:
                break  # Oldest messages dropped

        # 4. Reverse back to chronological order
        for msg in reversed(included):
            messages.append({"role": msg.role, "content": msg.content})

        return messages
```

### Testing Strategy

- **Unit tests for ContextBuilder:**
  - Short item content + short conversation → everything included
  - Long item content → truncated to cap, truncation note appended
  - Long conversation → oldest messages dropped, newest preserved
  - Empty item content → system prompt only, no content section
  - Single very long user message → truncated if needed, never dropped
  - Token budget math: system + item + conversation + reserved = model limit
  - Different models use correct context window sizes

## Milestone 2: Chat Backend

### Goal & Outcome

After this milestone:

- `POST /ai/chat` — streaming endpoint that accepts conversation history + item context
- Uses `ContextBuilder` to manage token budgets
- Returns SSE stream of assistant response chunks
- Cost tracking works for streaming (using `completion_cost` after stream consumption)

### Implementation Outline

#### 1. Request/response schemas

```python
class ChatMessage(BaseModel):
    role: str       # "user" or "assistant"
    content: str

class ChatRequest(BaseModel):
    messages: list[ChatMessage]
    context: str | None = None    # current item content
    entity_id: str | None = None  # for audit/logging
```

#### 2. Streaming endpoint (`backend/src/api/routers/ai.py`)

```python
@router.post("/chat")
async def chat(
    data: ChatRequest,
    current_user: User = Depends(get_current_user),
    limits: TierLimits = Depends(get_current_limits),
    llm_api_key: str | None = Depends(get_llm_api_key),
):
    config = llm_service.resolve_config(AIUseCase.CHAT, llm_api_key)

    context_builder = ContextBuilder(config.model, settings)
    messages = context_builder.build_messages(
        system_prompt=CHAT_SYSTEM_PROMPT,
        item_content=data.context,
        conversation=data.messages,
    )

    stream = await llm_service.stream(messages, config)

    async def event_generator():
        full_response = []
        async for chunk in stream:
            delta = chunk.choices[0].delta.content
            if delta:
                full_response.append(delta)
                yield f"data: {json.dumps({'content': delta})}\n\n"
        yield "data: [DONE]\n\n"

        # Track cost after stream completes
        cost = completion_cost(
            model=config.model,
            prompt="".join(m["content"] for m in messages),
            completion="".join(full_response),
        )
        await track_cost(redis, current_user.id, AIUseCase.CHAT, config.model, config.key_source, cost)

    return StreamingResponse(event_generator(), media_type="text/event-stream")
```

#### 3. Chat system prompt

Stored in `llm_prompts.py` alongside suggestion prompts. Instructs the LLM:
- You are a helpful assistant for a bookmarks and notes app
- The user's current item content is provided as context (if available)
- Be concise and direct
- When the content is truncated, acknowledge the limitation if the user asks about parts you can't see

### Testing Strategy

- **Backend:**
  - Streaming endpoint returns valid SSE format
  - Context is included via ContextBuilder (not raw)
  - Conversation history is passed through correctly
  - Empty messages list → 400 error
  - Rate limiting enforced (each chat request = one AI operation)
  - BYOK key passed through to LiteLLM
  - Cost tracking fires after stream completes
  - Client disconnect mid-stream → cost for consumed tokens still tracked (best-effort)

## Milestone 3: Chat Frontend

### Goal & Outcome

After this milestone:

- A chat sidebar panel on item detail pages
- Streaming responses with real-time token display
- Ephemeral conversations (cleared on navigation)

### Implementation Outline

#### 1. Chat sidebar component

A new right sidebar panel (similar to history sidebar) that appears on item detail pages:

- Toggle with a keyboard shortcut (TBD) or button in the toolbar
- Chat message list (scrollable, auto-scrolls to bottom on new messages)
- Text input at the bottom with send button (Enter to send, Shift+Enter for newline)
- Streaming: assistant messages update in real-time as SSE chunks arrive
- Loading indicator while waiting for first chunk
- Conversation messages stored in component state (ephemeral)
- Current item's content automatically included as context in each request

#### 2. SSE client

Use `fetch` with `ReadableStream` for SSE consumption (not `EventSource`, which doesn't support POST or custom headers):

```ts
const response = await fetch('/ai/chat', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`,
    ...(aiStore.apiKey ? { 'X-LLM-Api-Key': aiStore.apiKey } : {}),
  },
  body: JSON.stringify(chatRequest),
})

const reader = response.body.getReader()
const decoder = new TextDecoder()

while (true) {
  const { done, value } = await reader.read()
  if (done) break
  const text = decoder.decode(value)
  // Parse SSE lines, update message state
}
```

#### 3. Chat message rendering

- User messages: right-aligned, styled differently from assistant messages
- Assistant messages: left-aligned, supports markdown rendering
- Reuse existing markdown rendering approach for assistant messages

#### 4. Chat store (`stores/chatStore.ts`)

Zustand store for chat state:
- `messages: ChatMessage[]` — current conversation
- `isStreaming: boolean` — whether a response is in progress
- `sendMessage(content: string, itemContent?: string)` — appends user message, calls API, streams response
- `clear()` — resets conversation (called on navigation)
- `abort()` — cancels in-flight request via AbortController

### Testing Strategy

- **Frontend:**
  - Sending a message adds it to the conversation and triggers API call
  - Streaming response updates assistant message in real-time
  - Chat sidebar opens/closes correctly
  - Context from current item is included in requests
  - Navigation away clears conversation
  - Abort: clicking stop or navigating away cancels the stream
  - Error handling: network error → error message in chat
  - Error handling: 429 rate limit → appropriate message
  - BYOK header included when configured

## Milestone 4: Selection Actions via Chat

### Goal & Outcome

After this milestone:

- Users can select text in the editor, press Cmd+/, and choose AI actions (e.g. "Improve writing", "Summarize", "Explain")
- The action opens the chat sidebar (if not already open) and sends a pre-formatted message
- The user can iterate on the result conversationally

### Implementation Outline

#### 1. Backend: transform endpoint

The transform endpoint still exists as a convenience for non-streaming, single-shot transforms. But the frontend routes results through chat.

```
POST /ai/transform
```

```python
class TransformRequest(BaseModel):
    text: str                    # selected text
    action: str                  # "improve" | "summarize" | "explain"
    context: str | None = None   # surrounding text for better results

class TransformResponse(BaseModel):
    result: str
```

However, the primary flow is: the command menu sends a chat message like `"Improve the following text:\n\n{selected_text}"` and the response streams into the chat sidebar. The dedicated `/ai/transform` endpoint is available for cases where a non-streaming single-shot response is preferred (e.g. future inline replace), but the initial implementation uses chat.

#### 2. Command menu integration

Add AI commands to the existing `editorCommands.ts`:

```ts
// New section: "AI" (only shown when selection is non-empty and AI is available)
{ id: 'ai-improve', label: 'Improve writing', section: 'AI', icon: SparkleIcon }
{ id: 'ai-summarize', label: 'Summarize', section: 'AI', icon: ... }
{ id: 'ai-explain', label: 'Explain', section: 'AI', icon: ... }
```

These commands are conditionally included based on:
- User has a non-empty selection in the editor
- User's tier supports AI (or BYOK is configured)

#### 3. Chat integration flow

When the user selects an AI command:

1. Capture the selected text
2. Open the chat sidebar (if not already open)
3. Construct a user message based on the action:
   - Improve: `"Improve the following text:\n\n> {selected_text}"`
   - Summarize: `"Summarize the following:\n\n> {selected_text}"`
   - Explain: `"Explain the following:\n\n> {selected_text}"`
4. Send the message via `chatStore.sendMessage()`
5. The response streams into chat — user can follow up, ask for changes, etc.

This approach has two advantages over standalone UI:
- **No throwaway components** — no custom popover or inline replacement widget to build
- **Iterative** — user can say "make it more concise" or "actually keep the original tone" without starting over

### Testing Strategy

- **Frontend:**
  - AI commands appear in command menu only when text is selected
  - AI commands hidden when no selection or non-Pro without BYOK
  - Selecting "Improve" opens chat sidebar and sends formatted message
  - Selected text is included in the chat message
  - Response streams normally in chat
  - User can send follow-up messages to iterate
  - Loading state shown during API call
  - Error handling: API failure → error message in chat

## Cross-Cutting Concerns

### Context management evolution

The token-budget approach in this plan is intentionally simple — hard cap on item context, drop oldest messages. Future improvements (each could be its own plan):

- **Semantic chunking:** Instead of truncating from the front, split content into sections and include the most relevant ones based on the user's question
- **Conversation summarization:** Instead of dropping old messages, summarize them into a condensed message that preserves key context
- **Cross-item RAG:** Let the user ask questions across all their content, using embeddings to retrieve relevant items (builds on the pgvector-embeddings plan)

### Cost implications

Chat is the most expensive AI feature because:
- Each message resends the full conversation history + item context
- Streaming responses can be long
- Users may have many back-and-forth exchanges

The token budget system bounds per-request cost, but total cost per conversation scales linearly with turns. The daily cron flush (from the LLM Integration plan) provides visibility into chat-specific costs via the `use_case=chat` dimension.
