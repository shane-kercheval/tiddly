# Auth0 ↔ Clerk Capability Ledger

A living record of the functional differences between Auth0 and Clerk as discovered during Tiddly's migration ([plan](implementation_plans/2026-07-02-clerk-migration.md)). Each entry opens with **What** — plain-language context explaining what the capability is and why it matters here — followed by how each provider handles it, what migrating took, a **gained / lost / neutral** verdict, and any gotchas. Entries also record "possible on both providers, but…" nuances and the places we deliberately diverged from Clerk's own migration guidance. This is the blunt raw record; polished writeups get distilled from it, not the other way around.

**Status**: seeded 2026-07-02, before implementation started. Migration-effort notes are estimates until the relevant milestone completes.

**When to update this file** (for whoever — human or agent — is working the migration): update it whenever a finding surfaces or a claim changes, not only at milestone boundaries. Concretely: each milestone's Definition of Done requires updating it; any new fact, correction, or "possible on both, but…" nuance discovered mid-work gets folded into the relevant entry as it's learned; and when an item in the Open Questions table below gets answered, write the answer into the entry and flip its row per the marker convention documented there. Claims about either provider follow the plan's external-claims rule: verified against primary sources or listed under Open Questions — never asserted from memory or from secondhand summaries.

**Provenance — how to read each claim's basis** (be precise about what was *done* vs. *read*): a dated marker like *(confirmed 2026-07-06 …)* means verified empirically against our own dev instance — through the Clerk CLI or by minting and decoding a real token; the consolidated record is the **"M0 verification log"** in the Clerk-CLI entry. A "verified against \[doc]" note means checked against a primary source but not necessarily exercised in our instance. Anything not yet pinned either carries an `[OPEN]` marker into the Open Questions table or is dated research pending verification. Where an entry describes provider behavior we have **not** exercised, it says so rather than implying we tested it.

## Identity & user migration

### User export and password hashes
- **What**: To move users to a new auth provider without forcing everyone to reset their password, you need the password *hashes* (the one-way-encrypted form providers store) exported from the old provider. How hard a provider makes that export determines how painful it is to leave them.
- **Auth0**: Everything about your users exports self-serve *except* password hashes. Those require a support ticket (the magic words are "I would like to obtain an export of my tenant password hashes"), the turnaround is roughly a week, and support may require a paid tenant before releasing them.
- **Clerk**: The equivalent export, hashes included, is self-serve. If we ever leave Clerk, this step won't exist.
- **Alternative — skip the hashes entirely**: Clerk's user-creation API accepts `skip_password_requirement: true`, creating the user with no password at all (allowed as long as password isn't the instance's *only* sign-in method). Each affected user then sets a new password or uses a passwordless/social method on their first login. The trade is one reset event per password user versus a week of waiting on the ticket: with few password users the skip path usually wins; at scale the ticket earns its cost. In every no-password path, first-login identity is verified by proof of inbox ownership — Clerk emails a one-time code automatically (through the forgot-password flow, or through email-code sign-in if enabled); the application sends nothing. One inference worth noting: Clerk forbids the skip flag when password is the instance's *only* sign-in method, which suggested the forgot-password flow might not serve accounts that never had a password — resolved (question 10, confirmed 2026-07-11 in the M2 rehearsal): the question never arises in practice, because the hosted sign-in flow doesn't offer a password prompt to a passwordless account at all — with `email_code` enabled as a sign-in strategy it goes straight to "enter the code we emailed you," and the account stays passwordless (`password_enabled: false`) unless the user later sets a password themselves. (Which path Tiddly chose, and why, lives in the plan — M0 operator steps and M2.)
- **Verdict**: Gained — Clerk's self-serve export means a future exit from Clerk would be easier than an exit from Auth0.
- **Gotcha**: Whichever path a migration chooses, it must be decided before the import work is scheduled, because the ticket path inserts a week-long external wait that nothing else can absorb. If a ticket is filed at all, file it on day one so the wait overlaps with other work.

### Password import
- **What**: The receiving provider has to accept those exported hashes directly, or the export was pointless — this is what lets users' existing passwords keep working after the switch.
- **Auth0 → Clerk**: Clerk's user-creation API accepts `password_digest` with `password_hasher="bcrypt"`, which is exactly the format Auth0 uses (per Clerk's `CreateUser` API reference). An imported user would sign in with their old password and never know anything changed.
- **How verified / exercised**: doc-level only — **Tiddly does not use this path.** With one password user, M0 chose passwordless import (`skip_password_requirement`, which *was* exercised — see the M0 verification log) over a hash export. So `password_digest` is recorded as a capability, not something we tested.
- **Verdict**: Gained (for a migration that needs it). This single API parameter is most of the reason a password migration can be invisible to users — though we sidestepped it entirely.

### Social (Google) account transfer
- **What**: Users who sign in with Google have no password to migrate — their "credential" is a consent grant living at Google, and no provider can export another provider's grants. The question is how a migrated user's first Google sign-in gets attached to their imported account instead of creating a duplicate account.
- **Both providers**: OAuth grants are consent-bound and non-exportable. This is inherent to OAuth, not a vendor limitation.
- **Clerk**: Handles it through account linking by verified email: import the user with their email address marked *verified*, and when they later sign in with Google (returning a matching verified email), Clerk silently attaches the Google identity to the existing account.
- **Confirmed empirically (2026-07-11, M2 rehearsal)**: a user imported with a verified email clicked "Continue with Google" on the hosted sign-in as their first-ever Clerk interaction — no create-account step, no verification prompt; afterwards the instance's user count was unchanged and the Google identity appeared as an `oauth_google` external account on the same user id.
- **Verdict**: Neutral on mechanism, good ergonomics in practice.
- **Gotchas**: The import must mark emails as verified, or Clerk will prompt or block instead of linking silently. Separately: Auth0 treats each login method as its own user unless someone explicitly linked them, so the same human can appear twice in the Auth0 export (once as `auth0|...`, once as `google-oauth2|...`) with the same email. The import script must collapse those into one Clerk user, keyed by whichever identity Tiddly's database actually stores.

### Multi-identity / account linking (architectural)
- **What**: One human should be one account even if they sometimes log in with a password and sometimes with Google. Where that mapping lives — in your own database or inside the provider — shapes your schema.
- **Auth0**: Each login method is a distinct user (`sub`) unless explicitly linked via API calls. Tiddly had a designed-but-unbuilt plan (`future-identities.md`) to own this mapping ourselves with a `user_identities` table.
- **Clerk**: Linking happens inside the provider. One Clerk user can hold a password, a Google login, and more, while presenting a single stable `user_id` to our database.
- **Verdict**: Gained — an entire planned migration was deleted from our roadmap because the provider absorbs the problem. Probably the most underrated win of the switch.

### Divergences from Clerk's own migration guide (deliberate)
- **What**: Clerk publishes a recommended Auth0-to-Clerk migration path (https://clerk.com/docs/deployments/migrate-overview). We deviated from it in two places, and the reasons matter: the vendor's guide optimizes for large customers, and following it blindly would have left us with worse architecture.
1. **We skipped the `external_id` session-token aliasing.** The guide suggests configuring the session token to emit `{{user.external_id || user.id}}` so a backend can keep keying on legacy Auth0 IDs indefinitely. That exists for teams with large userbases and backend code they can't safely change. We control our backend and have a handful of users, so we do a clean swap of the lookup column instead. The old Auth0 ID is still stored in Clerk's `external_id` field, but purely as an audit breadcrumb — the application never reads it.
2. **We skipped trickle migration.** The guide describes running both providers in parallel and migrating each user when they happen to log in. That machinery pays off for big fleets. At our scale the right shape is: bulk import, a window where the backend accepts tokens from both providers, one coordinated client flip, a soak period, then decommission.

### Import rehearsal (M2) — what running against real production data caught
- **What**: The M2 import script was rehearsed end-to-end (2026-07-11): the real production Auth0 export, dry-run against a restored production Postgres backup, then a wet run (synthetic fixture) that created real users on the dev instance, backfilled, and re-ran idempotently. Everything below was invisible until real data hit the preflight — the argument for never letting the production run be the first full-fidelity run.
- **Databases accumulate identity-provider orphans silently.** 5 of 16 production user rows referenced Auth0 identities that no longer exist (early test accounts deleted from the tenant; the rows stayed). Two of them were *live* — PAT-authenticated security-test accounts whose PATs are used by the deployed penetration tests, identities long gone. Nothing in normal operation ever surfaces this drift; a migration import is where it becomes visible. The script's hard-fail-on-unmatched-rows design turned it into a legible operator decision (delete the dead rows; hand-link the PAT-only accounts to Clerk operator users) instead of a silent mis-import. Any team migrating providers should expect orphans and budget an investigation step.
- **`skip_password_requirement` is needed for *every* Backend-API create on a password-required instance — social-only users included.** The plan assumed only the password user needed it; the rehearsal's first create without the flag was rejected (`form_data_missing: ["password"]`). Caught on the dev instance instead of mid-cutover — the import now passes it unconditionally (no imported user gets a password, per the no-hash-export decision).
- **The self-serve export, in practice**: `auth0 api get "users?per_page=100&include_totals=true"` via the auth0 CLI — a repeatable command (re-run for the fresh M6a export) returning the documented user object with everything the import needs (`user_id`, `email`, `email_verified`, `identities`). No dashboard-extension clicking, and the `include_totals` envelope lets the script hard-fail on accidentally truncated/paginated exports.
- **Effort note for the writeup**: script + exhaustive decision-logic tests + full rehearsal (backup restore, dry-run, orphan investigation, wet run, idempotency re-run, first-sign-in and Google-linking verification) was one working session; the Clerk-side surface (`users.create` with `email_address_identification_status`, `external_id`, list pagination) held no surprises beyond the password-flag default above.

### Session/token model
- **What**: The core mechanics of how a logged-in browser proves its identity on every API call — how long tokens live, who refreshes them, and what the backend checks. This is the deepest architectural difference between the two providers.
- **Auth0**: Access tokens live for hours and are scoped to an API "audience"; the browser also holds a rotating refresh token and the SPA manages refreshing explicitly.
- **Clerk**: Session tokens are JWTs that live about 60 seconds and are refreshed automatically by Clerk's JavaScript via a cookie. The client never holds a refresh token for web sessions. Instead of an audience check, the backend checks `azp` ("authorized party" — which origin the token was issued to); Clerk session tokens carry no audience claim at all.
- **Migration effort**: The backend's claim validation changes shape (M1); the SPA swaps its interceptor's refresh logic for a simple `getToken()` call.
- **Verdict**: Gained, with one availability trade. The gains: the app *permanently deletes* its token-lifecycle machinery (the detect-expiry/refresh/retry/dedupe choreography — code that no longer exists can't have bugs); a tab that sat idle overnight transparently gets a fresh token instead of Auth0's expired-cached-token dance; and stolen tokens are near-worthless while revocation takes effect within about a minute (per the security-trade bullet below). The trade: web sessions survive a Clerk outage for only about a minute, versus hours of coasting on Auth0's long-lived tokens (see the Reliability entry).
- **The security trade behind each design**: a signed bearer token is valid until it expires, no matter what happens after it's issued — there is no practical way to recall one. Auth0's hours-long tokens therefore mean a stolen token works for hours. Clerk's one-minute tokens invert the trade: a stolen token is nearly worthless, and revocation is fast — banning a user or "sign out everywhere" stops the re-issuing, so every device is locked out within about a minute. The corresponding cost is availability: keeping a web session alive requires Clerk to be continuously reachable (see the Reliability entry).
- **End state worth naming once**: after the migration, three credential types coexist, with lifetimes matched to their clients — Clerk session tokens (web, ~60 seconds), Clerk OAuth access tokens (CLI and MCP clients, day-lived, since a terminal can't run background-refresh JavaScript), and app-managed PATs (long-lived, for scripts and headless use). All three are verified by the same backend.
- **Gotchas**: Any code assuming a token stays valid for hours breaks under 60-second tokens. Custom claims share roughly 1.2KB of budget inside a 4KB cookie. And the backend's auth cache must key on the user (`sub`), never on the raw token — tokens rotate every minute, so a token-keyed cache would never hit.
- **Clock-skew leeway (confirmed 2026-07-06 via the config schema)**: with 60-second tokens, a slightly-fast client clock can reject a still-valid token. Clerk exposes `session.allowed_clock_skew` (seconds), **default 5**, configurable 0–300. Important subtlety: this governs *Clerk's own* server-side validation — it is **not** inherited by our backend's independent PyJWT verification, which defaults `leeway=0`. So M1 must pass `leeway=5` explicitly (matching Clerk's value) or those spurious 401s ship anyway. (`session.lifetime` — the token TTL, default 60s, min 30 — is the same config object.)
- **`azp` is conditional, not universal (confirmed 2026-07-06 by minting a real token)**: `azp` (the requesting web origin) is present only on **browser-origin** tokens. A token minted via the Backend API carried **no `azp`** at all — and by the same logic, native-SDK (iOS) and OAuth tokens won't either. This differs from Auth0, whose `aud` was always present. So the backend rule must be "if `azp` present → it must be in the allowlist; if absent → tolerate," never a blanket requirement (which would reject every non-browser client). Safe because absence can't be forged onto a signed browser token. (This is the concrete basis for open question 11's likely answer.)
- **M1 implementation experience (2026-07-10 — dual-accept built and green)**: the backend now verifies both issuers, routing on the token's unverified `iss` (dispatch only; the selected verifier re-enforces everything). Effort data point for the writeup: the whole backend dual-accept — issuer routing, a second JWKS verifier with the azp rule and `leeway=5`, provider-neutral user resolution with per-issuer JIT-create flags, a DB CHECK identity invariant, cache re-keying, and ~60 new tests — was **one working session**, because Clerk session tokens are ordinary RS256 JWTs against a JWKS: the existing PyJWT pattern absorbed them with a parameterization, not a new stack. The azp check is ~4 lines; the leeway is one argument. What actually cost time was *our* migration-window machinery (JIT gates, transitional cache segments), not anything Clerk-shaped. Verified by `make backend-verify` (full suite, 3,283 tests) with real-signature test fixtures (RSA test keypair + mocked JWKS), not by decode-stubbing. **Then verified live (2026-07-11)** against the running local stack with real credentials from both providers: a dev-tenant Auth0 browser login JIT-created a user (namespaced email claims resolved, consent flow clean), a real Clerk dev-instance session token (minted via the Backend API) authenticated and JIT-created a Clerk-keyed row, and — after simulating M2's backfill with one SQL update — an Auth0 token and a Clerk token resolved to the **same `users.id`**, with the Clerk request passing the consent the Auth0 session had recorded. The dual-accept promise, demonstrated end to end.
- **M3 implementation experience (2026-07-12 — frontend flipped in code; production deploy held)**: the promised deletion happened — the SPA's detect-expiry/refresh/retry/dedupe choreography is gone, replaced by `getToken()` per request (the shared refresh-promise survives only to dedupe concurrent forced mints on 401 retry). Two findings for the writeup: (1) the session-expiry UX contract (in-place modal, popup OAuth, auto-retry of the parked request) cost more design care than the SDK swap itself — the swap was a one-file change thanks to the seam; the *product work* the short-session model demands is where the effort lives. (2) A subtle desync class surfaced in review: the client can believe it's signed in while the backend rejects its tokens (ban, clock skew, mis-configured verification) — any auto-retry logic keyed on "am I signed in now?" rather than "did a sign-in just happen?" silently loops against the backend. Applies to any short-token provider, not just Clerk.

### Custom claims (getting email into the token)
- **What**: The backend wants the user's email inside the verified token itself so it doesn't need an extra lookup on every request. Providers differ wildly in how much machinery this takes.
- **Auth0**: Required a deployed post-login "Action" (JavaScript running inside Auth0), namespaced claim names like `https://tiddly.me/email`, and an environment variable (`AUTH0_CUSTOM_CLAIM_NAMESPACE`) that our settings validation demanded even on cron services that never touch auth.
- **Clerk**: Shortcode-based claims, plain claim names, no namespace, no deployed code. The exact shortcodes are now confirmed against the dev instance: `email` → `{{user.primary_email_address}}`, `email_verified` → `{{user.email_verified}}` (verified 2026-07-06, set via the CLI — see the config-as-code entry). Beyond the dashboard's claims editor, the claims are settable as code (`clerk config patch` against `session.claims`), which is how ours were configured.
- **Verdict**: Gained. An entire category of configuration machinery — the Action code, the namespace variables, the validator plumbing — disappears at decommission.
- **Gotcha — the drift risk, now materially smaller than first assessed**: the original concern was that this config lives in a dashboard, outside version control, discoverable only when it breaks. Clerk's config-as-code CLI changes that: `clerk config pull` emits the whole instance config (including `session.claims`) as JSON that can be committed and diffed, and `clerk config patch/put` applies it back — so the configured claims now live in the repo as reviewable code (`clerk/config.dev.json`), not just documented in prose. The residual risk is only that someone changes the live instance out-of-band without re-pulling — mitigated by the drift-check in `clerk/README.md`. Also note the session-token claim-size budget (~1.2KB inside a 4KB cookie, per the session/token entry): irrelevant for an email address, a real constraint if more claims accumulate later.

### CLI auth: device authorization flow — the headline gap
- **What**: How a terminal program (`tiddly login`) logs a user in. The relevant standard, "device flow" (RFC 8628), works by showing the user a short code to enter in a browser *on any device* — which makes it the only flow that works when the terminal is on a remote SSH machine with no browser of its own.
- **Auth0**: Supports device flow, and our Go CLI is built on it.
- **Clerk**: Does not support it. Their OAuth server offers only the authorization-code and refresh-token grants; Clerk's own blog post about CLI auth explicitly scopes device flow out; and it sits on their public feedback roadmap as a backlog idea with no commitment.
- **Migration effort**: A full rewrite of the CLI's login flow (M4) to the pattern `gh auth login` uses: authorization code + PKCE, with the browser redirecting to a temporary listener on `127.0.0.1`. The keyring storage and token-refresh plumbing survive.
- **Alternatives considered**: (a) authorization code + PKCE with a loopback listener — adopted; the industry-standard CLI pattern. (b) Falling back to app-managed Personal Access Tokens for terminals where the loopback can't work — adopted for the SSH/headless case. (c) Hand-building a device-flow-like "paste this code" experience on top of Clerk's sign-in tokens (server-generated one-time tokens) — rejected: it amounts to implementing and securing your own device flow, which is precisely the machinery an auth provider should own.
- **Verdict**: Lost, but narrowly. The loss is specific to one scenario — a terminal on a machine with no browser (SSH into a remote box), where interactive login is no longer possible and a pasted PAT takes its place. For the majority case — a developer at their own desktop — the replacement is actually *smoother* than device flow: no code to read and re-type; the browser round-trip completes on its own.
- **Gotcha**: This is the single hardest gap for any CLI-first product migrating from Auth0. If Clerk ever ships device flow, this entry should be updated.

### OAuth provider capability (Clerk as authorization server)
- **What**: Beyond logging users into your own app, a provider can act as a full OAuth server, letting *third-party* clients — our CLI, or MCP clients like ChatGPT — request access to your API with the user's consent. This is the machinery that powers both the CLI rewrite and MCP OAuth.
- **Auth0**: A more complete OAuth authorization server, full stop — custom scopes (finer-grained consent), the `client_credentials` grant (machine-to-machine without a user), device flow (its own entry above), and dynamic client registration (the pre-migration MCP OAuth design, Appendix A of the plan, was written against Auth0's DCR — with the caveat of the 10-application free-tier entity limit).
- **Clerk**: Covers the subset most products need — authorization code with PKCE, refresh tokens that never expire (verified against Clerk's OAuth implementation doc), dynamic client registration, OAuth access tokens issued as JWTs by default since January 2026 (verifiable locally without calling Clerk) — with better ergonomics: a customizable/self-hostable consent screen and active first-party investment in the MCP auth spec.
- **Clerk's gaps**: No custom OAuth scopes — a consenting user grants all-or-nothing access. That matches our unscoped PATs, so nothing regresses for us, but finer-grained consent isn't available if we ever want it. No `client_credentials` grant (Clerk sells proprietary machine-to-machine tokens instead), which we don't need.
- **Verdict**: Mixed, and worth stating plainly: on capability breadth this area is a *loss* — Auth0 is the stronger pure OAuth server (scopes, client_credentials, device flow). Clerk covers everything Tiddly actually uses, with nicer defaults. A product that needs scoped third-party consent or standards-based M2M today would weigh this section very differently.

### PATs vs Clerk API Keys — a corrected first-pass analysis
- **What**: Tiddly has a home-built Personal Access Token system — the `bm_` tokens users create for scripts, the CLI, and MCP configurations. Clerk sells a direct managed equivalent called API Keys (GA April 2026). The question was whether to keep ours or adopt theirs.
- **Decision**: Keep our system untouched through the migration, then spike Clerk's product afterward. The spike must answer: does their key verification require a live call to Clerk or work locally; can we cap keys per user (we enforce tier quotas today); does the verified result expose enough to keep our audit trail; and does any import path exist for existing key hashes (see the migration-path gap below). If those come back clean, deleting our `token_service.py` and letting Clerk own credential storage is a legitimate simplification.
- **Migration-path gap — product feedback worth delivering to Clerk**: there is no documented way to import existing keys. Clerk's create endpoint *generates* a new secret and returns it once; nothing accepts a key or key-hash that already exists (verified against the API-keys docs; the Q8 spike confirms the negative definitively). There is no technical barrier: key verification is hash-and-compare, the stored hash is portable data, and **Clerk already built exactly this pattern for passwords** (`CreateUser` accepts `password_digest` + `password_hasher`, so migrated users' passwords keep working). An API-key equivalent — accept `{key_hash, algorithm, prefix}` — would let a service's existing keys keep working through adoption. As it stands, adopting Clerk API Keys invalidates every existing customer key, which for Tiddly is the single strongest deterrent and is plausibly a showstopper for most established services with homegrown key systems — precisely the segment the product would otherwise win. "The `password_digest` of API keys" is the one-line feature request.
- **Why wait**: Two reasons. Sequencing — our PATs cross the migration with zero churn, meaning every existing token and pasted config keeps working while everything else changes, which de-risks the whole project. And provider neutrality — the PATs cost zero migration work precisely because they were never coupled to Auth0, and moving them into Clerk trades that away.
- **What a first-pass evaluation got wrong** (recorded because it's useful to anyone assessing this product): the instinctive objections were "per-verification metering on a hot path" and "it couples PAT auth to Clerk availability." Neither survives scrutiny. The metering costs about $10 per million verifications. PAT traffic isn't clearly the hottest path, and it shrinks once MCP clients move to OAuth. And Clerk's OAuth JWTs verify locally anyway, so the availability gap is narrow and hinges on one unverified detail. The durable reasons to defer were sequencing and lock-in — not product weakness.

## Clients

### React SPA (Vite, non-Next.js)
- **What**: Tiddly's web frontend is React on Vite — not Next.js, which is Clerk's home turf. The question is whether the non-Next.js path is a first-class citizen.
- **Auth0**: Uses the **hosted login page** model — clicking "log in" redirects the user away to a page Auth0 hosts on its own domain, then back to the app afterward. Historically the security-conservative default (credentials are only ever typed on the provider's domain, and the provider controls that entire surface), at the cost of a jarring redirect and a login experience that never quite looks like your app. Operationally simple: works with zero DNS setup (a custom domain is optional polish).
- **Clerk**: First-class support for plain React — the login form is an **embedded component** (`<SignIn />`) rendered inside the app: no redirect, your page, your flow. Clerk's bet is that embedded can be made as safe as hosted while feeling native — a large share of what "better developer experience" concretely means in its pitch. Production requires DNS records on your own domain (Clerk's Frontend API served at `clerk.<your-domain>`; plain DNS — Cloudflare's proxying breaks validation). That requirement is load-bearing, not ceremony: serving Clerk's API from a subdomain of *your* domain makes the session cookie **first-party**, which is what keeps the 60-second-token refresh machinery reliable as browsers phase out third-party cookies.
- **M3 note (implemented)**: the swap used Clerk's modal components (`openSignIn`/`openSignUp`) rather than a routed sign-in page — mapping 1:1 onto the old "button → hosted page" call sites with no new routes, and the same modal pattern serves the session-expiry re-auth dialog. `ClerkProvider` wires into React Router via `routerPush`/`routerReplace`, so Clerk-driven redirects are client-side transitions. `useClerk().status` (`'error'` = clerk-js failed to init) is the init-failure signal a provider swap must re-wire — Auth0 surfaced this through its hook's `error` field; hardcoding it away regressed the recovery screen until review caught it.
- **Verdict**: Neutral to gained.
- **Gotcha**: The ecosystem is Next.js-first. New features (billing components, MCP helpers) land there before plain React, and the Python verification SDK is solid but thinner than the JavaScript ones. Expect to hand-roll occasionally.

### End-user account management (profile, password, and security self-service)
- **What**: Users expect to manage their own account — change their password, see where they're signed in, manage linked logins, delete their account. Somebody has to build that UI, and which somebody differs sharply between providers.
- **Auth0**: Ships no end-user account UI. Its own architecture docs state self-service profile management "must be implemented and hosted external to Auth0" — the app builds it against Auth0's APIs (Management API, or the newer My Account API). The practical consequence for Tiddly: none of it ever got built. As of the migration, Tiddly has no password-change UI, no session management, and no account deletion — each was a build-it-yourself project that never justified itself.
- **Clerk**: Ships it — the `<UserProfile />` component ("full-featured account management UI... profile and security settings") mountable inside the app, plus a hosted Account Portal. The exact section list (sessions/devices, connected accounts, authorized-OAuth-app revocation, account deletion) is enumerated in the dev instance rather than assumed — open question 12 [OPEN].
- **Migration effort**: Mount the component as a settings page (plan M3, step 9).
- **Verdict**: Gained — several user-facing capabilities Tiddly never built arrive nearly free. Alongside account linking, this is the clearest "the provider absorbs work you were never going to do" entry.
- **Gotcha — account deletion must not be exposed blindly**: if the component includes a delete-account section, a user deleting themselves in Clerk removes the *identity* while all their data sits orphaned in the app's database. Deletion is the one flow where the provider and the app's database must move together — via a **webhook** (Clerk calls a URL on our backend when a `user.deleted` event fires; the handler runs our existing cascade delete). Two requirements come with that: the endpoint must **verify Clerk's signature** (a webhook is inbound and forgeable — an unauthenticated "delete this user" endpoint is a serious hole), and the handler must be **idempotent** (webhook delivery is best-effort and can repeat). Until deletion is wired end-to-end this way, the section must be hidden or configured away — **and "hidden" only governs surfaces the app renders**: Clerk also hosts its own account pages (the Account Portal), so app-side hiding alone doesn't close the flow. That's why the plan makes the deletion webhook a cutover prerequisite (question 12 tracks whether the hosted portal can be disabled outright).
- **Gotcha — during a dual-provider migration window, provider-side security controls bind only their own provider's ceremony**: MFA, "sign out everywhere," and similar controls configured in the new provider do nothing against sessions and logins still flowing through the old one — the whole point of a dual-accept window is that the old ceremony keeps working. Any team running this kind of migration should either delay shipping such controls until decommission or ship them with an explicit "full effect begins when the old provider dies" caveat. (Tiddly's case: moot for MFA — deferred with the free-tier decision — but recorded because it generalizes; deletion is the one control given an app-side backstop that spans both providers, via tombstones on both identity columns.)

### Chrome extension (MV3)
- **What**: Tiddly's browser extension saves bookmarks through the API. Today a user authenticates it by creating a Personal Access Token in the web app and pasting it into the extension's options page. That friction exists because running an Auth0 login inside a Manifest V3 extension was painful — the PAT paste was pragmatism, not preference.
- **Clerk**: Ships an official extension SDK with a feature called Sync Host: the extension *shares the web app's session*, so a user signed in at tiddly.me has a signed-in extension with zero setup. It has a vanilla-JavaScript entry point (`createClerkClient` — no React required), a documented background-service-worker pattern that keeps the 60-second tokens refreshing while the popup is closed, and full support for development instances (`syncHost: "http://localhost"` against a locally running web app).
- **Verdict**: Gained — a capability with no comparably easy Auth0 equivalent. Planned as M7, shipping after the production cutover, with the PAT paste retained as a fallback.
- **Gotchas** (each verified against primary docs after review rounds caught errors in secondhand claims): the SDK is only consumable as an npm import, so a bundler is mandatory — our extension had no build step at all, making that genuinely new scope. The production `syncHost` value is the Clerk Frontend API domain (`clerk.tiddly.me`), not the web app's origin. The manifest needs the `cookies` permission — not incidental: Sync Host works by the extension *reading the web app's session cookie* from the sync-host domain, so cookie access is the mechanism itself, and "wants to read cookies for your site" is why store review and user re-approval prompts treat the update seriously. And Clerk's `allowed_origins` update replaces the whole array — fetch, merge, then write, or you clobber existing origins.

### iOS
- **What**: Tiddly's iOS app (a separate repository) authenticates via Auth0 today and needs its auth layer rewritten no matter which provider wins.
- **Clerk**: The native SDK (ClerkKit) reached v1 GA in February 2026, with native Google Sign-In (the OS-level account sheet rather than a browser bounce — better mobile UX) and prebuilt auth/profile views. There are no cookies in a native app, so the SDK holds the session in the app's secure storage and does the token-refresh work itself — the same job clerk-js does in a browser, in a different container; the Auth0.swift token/refresh/rotation bookkeeping gets deleted just like the web interceptor's did. The migration plan's dual-accept window exists partly so the iOS app can migrate on its own timeline without blocking the main cutover — the final Auth0 decommission is explicitly gated on iOS shipping.
- **Gotcha**: a native app has no web origin, so what the `azp` claim contains in iOS-minted session tokens — and what the backend's authorized-parties allowlist must therefore include — is unverified; open question 11 [OPEN]. This is the one place the backend could silently 401 all iOS traffic.
- **Where the detail lives**: the standalone guide for the iOS developer, `docs/ios-clerk-migration-guide.md`.
- **Verdict**: Neutral. Well supported on both sides, and a rewrite is unavoidable either way.

### MCP servers / AI-agent auth
- **What**: Tiddly runs two MCP servers so AI tools (Claude, ChatGPT, coding agents) can manage a user's content. ChatGPT and Claude Desktop's native connectors *require* OAuth — they have no way to accept a pasted bearer token — so whether the provider can act as the OAuth server for MCP determines whether those integrations are possible at all. Background and requirements live in the plan's Appendix A.
- **Both providers can do this** (the pattern is: a discovery endpoint on our servers plus dynamic client registration at the provider). An Auth0 version was fully designed before the migration. MCP OAuth alone is therefore not a reason to switch providers.
- **Clerk's deltas**: OAuth access tokens are JWTs by default (locally verifiable), the consent screen is customizable and self-hostable, and Clerk actively invests in the MCP auth spec. One open comparison point: Auth0's free tier caps total applications at 10, which open client registration could exhaust; how Clerk meters registered clients is open question 4 [OPEN].
- **Why a hard application cap is structurally wrong for open registration (the precise version of "the 10-app cap blocks this")**: DCR registration *granularity is the client's choice, not the server operator's* — the spec allows anonymous registration and says nothing about reuse. Observed MCP-ecosystem behavior spans the full range: one shared registration per platform (the optimistic case, under which 10 slots would survive), per-user or per-install registrations, and re-registration whenever a client loses its stored credentials (reinstall, state reset). That unbounded growth is the explicitly stated motivation for the MCP spec's Client ID Metadata Documents proposal, which aims to replace DCR. Under a hard cap shared with first-party apps (~6–7 spare slots on our Auth0 tenant), the feature's availability would depend on every third-party client staying polite forever, and the failure mode — cap reached, new connector setups silently fail, unfixable from our side — is invisible until it happens. A provider without a low entity cap removes the dependency entirely; it doesn't matter which registration behavior clients choose.
- **DCR guardrail philosophies differ (verified)**: Auth0 gates *who may register* (an access-control list in front of registration). Clerk leaves registration deliberately open and anonymous, and instead **enforces the consent screen whenever DCR is enabled — it cannot be disabled** (documented rationale: without consent, a malicious authorization URL could silently obtain a logged-in user's grant). Consequences: every agent connecting hits a human consent moment at least once — no silent onboarding, arguably the right default for a personal knowledge base — and because Clerk has no custom scopes, that one screen approves *everything*, concentrating the anti-abuse weight on its clarity (which is why the plan's M5 security review inspects exactly what it shows users). This also reframes question 4: with forced consent as the primary access control, registration metering is spam hygiene rather than access risk.
- **Verdict**: Gained in ergonomics; honest entry for "possible on both."

## Platform & operations

### Environments
- **What**: How each provider separates real users from the environment you develop and test against.
- **Auth0**: Fully independent tenants — we ran a dev tenant and a prod tenant with nothing shared.
- **Clerk**: One *application* with paired dev and prod *instances* (observed directly in M0 — creating the `Tiddly` app produced a dev instance, and `clerk apps list` shows the app/instance structure). Users and secrets do not transfer between instances, a `clerk deploy` command clones configuration from dev to prod, and dev instances work on localhost with no DNS setup.
- **Verdict**: Neutral, mildly gained — the config cloning and localhost-friendly development are conveniences Auth0 didn't offer.

### Tooling: the Clerk CLI and config-as-code
- **What**: How much of provider setup can be driven from the command line / version control versus clicking through a dashboard. This matters for reproducibility, for reviewing auth changes like code, and for letting an automated agent do the work.
- **Clerk**: Ships a first-class CLI (v1.5 as of 2026-07). It reads and writes an instance's *entire* configuration as JSON — `clerk config pull` (dump), `clerk config schema` (the typed schema), `clerk config patch`/`put` (apply), all with a `--dry-run` that validates and shows a before/after without mutating. It also does `clerk apps create`, `clerk env pull` (keys into a local env file), `clerk api` (authenticated Backend-API client), and `clerk deploy` (dev→prod promotion). Two notable design choices: authentication is a one-time browser login (`clerk auth login`), and there is an explicit `--mode agent` plus a bundled agent skill — the CLI is deliberately built to be driven by AI agents, not just humans.
- **What it covered for us (M0)**: creating the application, and configuring the dev instance's auth strategies and session-token claims (`session.claims`) — all as code. The *only* human-in-the-loop step was the initial browser login; everything after was scriptable. The config schema itself answered several open questions directly (claim shape, clock-skew leeway, session lifetime, the `auth_access_control.sign_up_mode` restriction mechanism) — reading the schema replaced guessing.
- **What it does *not* cover**: `clerk enable` is limited to orgs/billing (auth strategies are `config` keys, not `enable` commands); and real social-provider credentials (Google/Apple client id + secret for production) still require the provider-side OAuth setup, not just a Clerk config write.
- **M0 verification log — what was confirmed, and how (all agent-driven through the CLI, 2026-07-06)**:
  - *Read from the config schema/pull* (`clerk config schema`/`pull`): session-token claim shape; clock-skew leeway (`session.allowed_clock_skew`, default 5); session token TTL (`session.lifetime`, 60s); sign-up-restriction key (`auth_access_control.sign_up_mode`); email/password + Google on by default, MFA/passkeys off.
  - *Minted and decoded a real session token* (Backend API: create user → `POST /sessions` → `POST /sessions/{id}/tokens`, then PyJWT against the live JWKS): signature + `iss` + `exp` verify with `leeway=5`; `email`/`email_verified` claims resolve (the latter a real boolean); `sub` = Clerk user id; 60s lifetime; **`azp` absent** on Backend-API tokens (see the session/token entry's `azp` gotcha).
  - *Backend-API user creation*: `skip_password_requirement: true` creates a passwordless user cleanly (de-risks M2); a BAPI-created email is marked `verified` by default (the M2 over-vouching footgun — the import must set verification from the export, not the default).
  - Deferred (not yet verifiable in M0): the OAuth-app access-token JWT shape (needs M4's loopback flow); the exact `azp` *value* on real browser tokens (M1 local testing).
- **Auth0 comparison (verified 2026-07-06 against Auth0's own docs — earlier draft of this entry was wrong on both counts)**: config-as-code is **not** a Clerk-exclusive capability. Auth0 ships an **official** Deploy CLI (`auth0/auth0-deploy-cli`) that exports/imports tenant configuration as files, and an **officially supported** Terraform provider whose `plan`/`apply` workflow is a dry-run equivalent. So on config-as-code and dry-run, this is closer to parity than advantage. Clerk's *actual* edge is narrower but real: a **single unified CLI** with inline `config patch`/`--dry-run` **plus an explicit agent mode** (`--mode agent` + a bundled agent skill), i.e. agent-native design — versus Auth0's capability split across the Deploy CLI, the separate `auth0` CLI, and Terraform, none agent-oriented.
- **Verdict**: Gained — but on *ergonomics and agent-native workflow*, not on exclusive config-as-code capability. It made M0 fast, reviewable, and almost fully automatable, and it shrinks the dashboard-drift risk called out in the custom-claims entry (the dev config is now committed as `clerk/config.dev.json`).

### Billing
- **What**: Tiddly has subscription tiers but zero billing code — `tier` is a plain database column, with no Stripe integration and no way for a user to pay. Clerk sells a billing product that could become that missing layer, which is part of why the migration was attractive.
- **Clerk Billing** (from Clerk Billing docs, mid-2026; not exercised — billing is deferred): Plans and features map directly onto our tiers. The user's plan is embedded in the session token (`pla`/`fea` claims), so server-side tier enforcement reads the already-verified JWT with zero extra calls. Stripe is the payment processor underneath; Clerk takes 0.7% on top of Stripe's fees.
- **Hard gaps**: No refunds through Clerk, USD only, no tax/VAT handling, no 3D Secure.
- **Decision**: Deferred, deliberately decoupled from the auth migration. The VAT gap could be disqualifying for European sales; when billing actually ships, evaluate Clerk Billing against Stripe Billing or Paddle with that gap front and center.

### Organizations / teams / B2B (future need — out of migration scope, but part of the original switch rationale)
- **What**: "Teams in Tiddly" would split into two layers. The **membership layer** is the generic bookkeeping every SaaS reinvents: org entities, user↔org memberships with roles, email invitations, multi-org switching. The **domain layer** is what membership *means* in the product: team-owned bookmarks/notes/prompts, shared-content permissions, per-org quotas and billing. A provider can absorb the first layer; the second is always the app's.
- **Clerk**: Ships the membership layer nearly whole — organizations, memberships, roles/permissions, invitation flows, plus prebuilt UI (org switcher, create-org, org-profile/member-management components, including on iOS). Architecturally the key integration point mirrors custom claims and billing: **the session token carries the active org and role**, so a backend that already scopes queries by `user_id` extends to `org_id` + role checks on the same verified-JWT path, with no extra provider calls. The enterprise glue on top: **self-serve SSO** (a customer's IT admin configures their own Okta/SAML without the vendor in the loop) and **SCIM directory sync** ("employee leaves the company → access dies automatically"), both shipped as of 2026 — these are the features that gate enterprise deals and cost quarters to hand-build. B2B billing supports per-seat plans.
- **Auth0**: Also has an Organizations feature (org entities, invitations, per-org login/branding), so the membership layer itself is not Clerk-exclusive. The differentiators are Clerk's prebuilt UI components, the self-serve SSO/SCIM layer, and B2B billing integration. A rigorous side-by-side is deferred until teams become a real roadmap item.
- **Costs to re-verify when this becomes real** [OPEN — no milestone; owner: Shane, when teams reach the roadmap]: B2B feature pricing (research from mid-2026 said a ~$100/month add-on plus per-organization pricing at scale) and the current feature-by-feature Auth0 comparison.
- **Verdict**: Deferred — but the shape of the win is the recurring pattern in this ledger: the provider absorbs generic, security-sensitive machinery (membership layer) so the product only builds what's actually its own (domain layer). Turning "build teams" from two hard projects into one is the realistic framing, not "Clerk gives you teams."

### Reliability
- **What**: When an auth provider goes down, users can't sign in. What is each provider's track record, and what is our exposure?
- **Clerk**: Two acknowledged major outages — September 2025, and March 2026 (26 minutes of errors, followed by a public postmortem committing to fixes).
- **Our exposure differs by credential type** (see the session/token entry): web sessions live on ~60-second tokens that Clerk must continuously re-issue, so a Clerk outage ends signed-in web sessions within about a minute. The longer-lived credentials ride through — CLI/MCP OAuth access tokens (day-lived) keep verifying locally against cached keys, and PATs don't involve Clerk at all. New sign-ins fail during an outage on any provider.
- **Honest asymmetry vs Auth0**: Auth0's hours-long tokens would let already-signed-in web users coast through a comparable outage; Clerk's short-token design trades that away for the revocation and stolen-token benefits described in the session/token entry. Same trade, seen from the failure side.
- **Verdict**: Recorded as an accepted risk at our current scale; revisit (e.g. tolerating slightly-stale tokens during a provider incident) only if it actually bites.

### Pricing
- **What**: What the provider costs as the user base grows.
- **Clerk** (verified against clerk.com/pricing, 2026-07): The free (Hobby) tier covers 50,000 monthly *retained* users and is production-ready, but excludes MFA and passkeys, shows "Secured by Clerk" branding, and **fixes session lifetime at 7 days** (weekly forced re-login). Pro is $25/month and lifts all four. Basic organizations (up to 20 members) are free on all plans; the enhanced B2B add-on is $100/month. Still cheaper than Auth0 at any scale Tiddly will see soon, and "retained users" is a friendlier billing metric than Auth0's monthly-actives.
- **Gotcha**: a third-party pricing survey claimed MFA and passkeys were free-tier; the primary pricing page says otherwise. Tier-gating claims age fast and vary by source — verify against clerk.com/pricing, dated. One gating fact the pricing page *doesn't* state: which tier the OAuth Applications feature needs — resolved (question 14): available on the free tier, confirmed 2026-07-10 by hitting `GET /oauth_applications` on our free-tier dev instance (normal empty-list response, no payment gate) plus the absence of any paid-plan badge on the OAuth-provider docs.
- **The session-lifetime regression (verified both sides; the sharpest free-tier pain)**: Clerk's free tier fixes sessions at a 7-day **absolute** maximum lifetime — expiry counts from sign-in regardless of activity, so every active user re-authenticates weekly. Which clients it hits: the web app and the extension's synced session expire at the same instant (via Sync Host they are literally the same session); an iOS session is subject to the same 7-day cap but runs its own clock from that device's sign-in. Which clients it does *not* touch: OAuth-application grants (the CLI's and MCP consumers' tokens — day-lived access tokens silently renewed by refresh tokens that never expire, per Clerk's OAuth docs) and app-managed PATs, which have no expiry at all. The net effect is mildly ironic: the CLI ends up with a longer-lived login than the web app. An inactivity-timeout mode exists but customizing either setting requires a paid plan in production (per Clerk's session-options docs). Auth0's comparable default is a 30-day **inactivity** refresh-token lifetime (Dashboard → Applications → [app] → Settings → Refresh Token Expiration) — an active user is effectively never logged out. Migrating Auth0-free → Clerk-free is therefore a real, user-visible session regression that only money fixes; any team evaluating this migration should price it in. Tiddly's decision — accept it at beta scale, upgrade when there are real users — lives in the plan's adoption register. The free-tier session cap is also publicly criticized (a Hacker News thread calls it out among "dark patterns in the free tier"; Clerk's response: fine for validating a product, upgrade when it matters).
- **The data-loss half of the regression is the app's bug, not Clerk's — and Clerk's model enables the fix**: what makes expiry *destructive* (rather than a one-click speed bump) is a frontend 401 handler that logs out, clears state, and navigates — a latent flaw under any provider that long sessions merely mask. The standard mitigation, adopted in the plan's M3: re-authenticate **in place** (sign-in modal over the untouched page, social login in a **popup** — Clerk supports `oauthFlow: "popup"`, verified — then auto-retry the failed request), a deterministic pre-expiry warning (the expiry time is client-knowable), and draft autosave as defense in depth. Handled this way, the weekly re-auth's real cost for a social-login user (virtually all of Tiddly's) is **one click** — the popup shows Google's account picker, since the user is already signed into Google in the browser — and password users retype a password into the in-page modal; that's the number that prices the regression honestly. On the popup itself: all major browsers, Safari included, *allow* popups opened synchronously from a user click — blocking is the edge case (an async gap between click and open breaking the gesture chain, or an aggressive third-party blocker), not the norm, and the draft covers even that. Ironic footnote: this graceful pattern is *easier* on Clerk than Auth0, whose redirect-based login destroys page state by construction.
